# Gin电商秒杀API服务技术规范文档

## 一、通用规范
### 1.1 术语规范
遵循系统设计文档中定义的术语，核心术语如下：
| 术语 | 定义 |
|------|------|
| 原子操作 | 不可中断的操作，如Redis的DECR命令，确保并发场景下数据一致性 |
| 削峰填谷 | 通过消息队列缓存瞬时高并发请求，异步消费处理，缓解后端压力 |
| QPS | 每秒查询率，衡量系统并发处理能力的核心指标 |
| 软删除 | 通过标记字段标识数据删除状态，而非物理删除数据 |

### 1.2 编码规范
- 开发语言采用Go 1.25.1，遵循Go官方编码规范
- 前端采用Vue 3 + Vite，遵循Vue官方编码规范
- 数据库操作使用GORM框架，遵循ORM最佳实践

## 二、架构规范
### 2.1 分层架构规范
系统严格遵循三层架构设计，各层职责边界清晰：
1. **数据层**：仅负责数据的存储与基础管理，不包含业务逻辑
   - MySQL：存储用户、商品、订单等核心业务数据，使用InnoDB引擎
   - Redis：负责缓存、限流计数、消息队列功能
2. **服务层**：封装核心业务逻辑，提供模块化服务能力，各服务间通过接口交互
3. **接口层**：基于Gin框架实现HTTP接口，处理请求与响应，集成必要中间件

### 2.2 模块划分规范
系统按核心业务域划分为以下模块，模块间低耦合设计：
- 用户模块：处理用户注册、登录、信息查询
- 商品模块：管理商品基础信息
- 秒杀活动模块：管理秒杀活动配置
- 秒杀模块：处理秒杀请求与库存扣减
- 订单模块：负责订单生成、存储与查询

## 三、数据存储规范
### 3.1 MySQL规范
#### 3.1.1 数据库配置规范
- 数据库版本：MySQL 8.0
- 字符集：UTF-8mb4
- 存储引擎：InnoDB（支持事务与行级锁）

#### 3.1.2 表结构规范
严格遵循系统设计文档定义的表结构，核心表结构如下：

**用户表（User）**
```go
type User struct {
   ID  uint   `gorm:"primaryKey" json:"id"`                         // 主键ID
   Username string `gorm:"size:50;uniqueIndex;not null" json:"username"` // 用户名(唯一)
   Password string `gorm:"size:100;not null" json:"-"`                 // 密码(密文，返回JSON时隐藏)
   Phone string `gorm:"size:20;uniqueIndex" json:"phone"`             // 手机号(唯一)
   CreatedAt time.Time `json:"created_at"`                          // 创建时间
   DeletedAt gorm.DeletedAt `gorm:"index" json:"-"`                  // 软删除标记
}
```

**商品表（Product）**
```go
type Product struct {
   ID    uint64  `gorm:"primaryKey" json:"id"`                     // 主键ID
   Name  string  `gorm:"size:100;not null" json:"name"`            // 商品名称
   Price float64 `gorm:"type:decimal(10,2);not null" json:"price"` // 商品价格
   Stock int     `gorm:"not null;default:0" json:"stock"`          // 总库存
   DeletedAt gorm.DeletedAt `gorm:"index" json:"-"` // 软删除标记
}
```

**秒杀活动表（Event）**
```go
type Event struct {
   ID        uint           `gorm:"primaryKey" json:"id"`             // 主键ID
   ProductID uint           `gorm:"not null;index" json:"product_id"` // 关联商品ID
   StartTime time.Time      `gorm:"not null" json:"start_time"`       // 活动开始时间
   EndTime   time.Time      `gorm:"not null" json:"end_time"`         // 活动结束时间
   DeletedAt gorm.DeletedAt `gorm:"index" json:"-"`                   // 软删除标记
}
```

**订单表（Order）**
```go
type Order struct {
   ID    uint      `gorm:"primaryKey" json:"id"`                    // 主键ID
   UserID uint      `gorm:"not null;index" json:"user_id"`             // 关联用户ID
   ProductID uint  `gorm:"not null;index" json:"product_id"`           // 关联商品ID
   OrderAmount float64 `gorm:"type:decimal(10,2);not null" json:"order_amount"` // 订单金额
   Status OrderStatus `gorm:"not null;default:0" json:"status"`          // 订单状态
   CreatedAt time.Time `json:"created_at"`                         // 创建时间
   DeletedAt gorm.DeletedAt `gorm:"index" json:"-"`                 // 软删除标记
}
```

#### 3.1.3 数据操作规范
- 所有删除操作采用软删除，通过`deleted_at`字段标记
- 关联查询使用GORM的关联查询功能，避免手动JOIN
- 批量操作使用GORM的批量处理API，减少数据库连接开销

### 3.2 Redis规范
#### 3.2.1 缓存键名规范
- 库存缓存：`seckill:stock:{product_id}`
- 用户抢购记录：`seckill:user:{product_id}:{user_id}`

#### 3.2.2 数据操作规范
- 库存扣减必须使用原子操作（DECR命令或Lua脚本）
- 秒杀相关缓存设置合理过期时间，用户抢购记录过期时间为86400秒
- 使用Redis作为消息队列时，采用`LPUSH`写入、`BRPOP`读取模式

## 五、核心业务实现规范
### 5.1 秒杀流程规范
1. 秒杀活动创建时，必须将商品库存同步至Redis，执行`SET seckill:stock:{product_id} {stock}`
2. 处理秒杀请求时，必须使用以下Lua脚本进行原子操作：
```lua
local stockKey = "seckill:stock:" .. KEYS[1]
local userKey = "seckill:user:" .. KEYS[1] .. ":" .. ARGV[1]
local stock = redis.call("GET", stockKey)

if not stock or tonumber(stock) <= 0 then
    return 0 -- 库存不足
end

if redis.call("EXISTS", userKey) == 1 then
    return 2 -- 已抢购
end

redis.call("DECR", stockKey)
redis.call("SET", userKey, 1, "EX", 86400)
return 1 -- 扣减成功
```
3. 秒杀成功后，立即返回结果，不等待订单写入MySQL
4. 订单信息通过Redis消息队列异步写入MySQL，确保主流程无阻塞

### 5.2 订单处理规范
1. 订单生成后，通过`LPUSH`写入Redis消息队列
2. 订单消费Worker使用`BRPOP`阻塞读取消息队列
3. 订单写入MySQL失败时，最多重试3次，仍失败则记录异常日志
4. 订单状态流转需符合业务规则，记录完整状态变更日志

## 六、部署规范
### 6.1 环境配置规范
- 后端框架：Gin (Go 1.25.1)
- 前端框架：Vue 3 + Vite
- 数据库：MySQL 8.0（端口3307）
- 缓存：Redis 7（端口6379）
- 消息队列：RabbitMQ 3-management（端口5672）
- 容器化：Docker + Docker Compose

### 6.2 部署流程规范
1. 先安装并初始化MySQL、Redis，执行SQL脚本创建表结构
2. Redis配置主从（集群部署时），启用RDB+AOF持久化策略
3. 编译Go项目生成可执行文件，配置config.yaml（包含数据库地址、Redis地址、限流参数等）
4. 启动API服务与订单Worker进程
5. 配置Nginx反向代理，集群部署时采用轮询负载均衡策略

## 七、性能与安全规范
### 7.1 性能优化规范
- 热点数据（商品、库存）必须通过Redis缓存，减少MySQL查询
- 订单创建采用异步写入模式，避免同步操作阻塞秒杀流程
- 接口层添加合理缓存策略，减少重复计算与查询

### 7.2 安全保障规范
- 库存扣减必须使用原子操作，杜绝超卖/少卖问题
- 订单消费实现重试机制，确保数据不丢失
- 所有敏感数据（如密码）必须加密存储
- 接口请求必须进行合法性校验，防止恶意攻击
