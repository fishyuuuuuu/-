# 基于 Gin 的电商秒杀 API 服务项目任务书

## 一、项目背景

在电商行业中，秒杀活动作为提升平台流量与用户活跃度的重要手段，常面临瞬时高并发请求的冲击。此类场景下，系统易出现“超卖”“少卖”、响应延迟甚至崩溃等问题，同时机器人刷单、恶意抢购等行为也会破坏活动公平性。为解决上述痛点，本项目拟开发一款基于 Gin 框架的高并发电商秒杀 API 服务，结合 Redis、消息队列等技术，实现库存精准控制、流量削峰填谷及恶意请求拦截，保障秒杀活动稳定、高效、公平开展。

## 二、项目目标

1.  功能目标：实现商品库存管理、秒杀请求处理、订单异步落库核心功能，配套用户认证、图形验证码、请求频次限制等拓展功能，杜绝超卖、少卖问题。
2.  性能目标：支撑瞬时 1000QPS 以上并发请求，接口平均响应时间≤200ms，Redis库存扣减原子性准确率100%。
3.  可靠性目标：服务可用性≥99.9%，消息队列消息投递成功率≥99.99%，订单数据最终一致性达标。

## 三、项目范围

### （一）核心功能

1.  库存管理：支持商品库存 Redis 预加载、基于 Lua 脚本的原子性扣减、库存实时查询。
2.  秒杀处理：接收用户秒杀请求，完成库存校验与扣减，触发订单异步处理流程。
3.  订单处理：库存扣减成功后，订单信息入消息队列，由 Worker Goroutine 消费并写入 MySQL。
4.  流量控制：基于令牌桶算法实现接口级限流，基于用户 ID/IP 实现频次限制。

### （二）非功能需求

1.  高可用性：Redis 采用主从架构，服务支持水平扩展，避免单点故障。
2.  可扩展性：模块间低耦合，支持后续新增秒杀商品类型、更换消息队列等需求。
3.  可维护性：代码符合 Go 语言规范，关键逻辑添加注释，提供完整开发与部署文档。

## 四、核心技术栈

1.  Web 框架：Gin
2.  数据存储：Redis（缓存）、MySQL（持久化存储）
3.  数据库操作：gorm
4.  消息队列：RabbitMQ
5.  其他：go-redis、amqp、Vue3等

## 五、任务分解与实施计划

### 第一周：项目准备与基础框架搭建

*   **Day 1-2**: 学习Gin框架基础（路由、中间件、绑定）、Redis基础命令。
*   **Day 3-4**: 搭建项目工程结构（`/config`, `/controller`, `/model`, `/service`, `/router`）。集成Gin、Viper（配置）、GORM（连接MySQL）。
*   **Day 5**: 实现用户模块的Model层（GORM结构体）和DAO层（数据库操作），完成用户注册/登录接口。
*   **目标**：完成项目框架搭建和用户基础功能。

### 第二周：商品与秒杀逻辑实现

*   **Day 6-7**: 实现商品和秒杀活动的Model和CRUD接口（增删改查，仅供管理员使用）。
*   **Day 8-9**: **实现同步秒杀接口（V1版）**。直接在Gin handler中完成：查库存 -> 扣MySQL库存 -> 创建订单。**这个版本会有严重的超卖和高并发问题**，但用于理解流程和后续对比。
*   **Day 10-11**: **引入Redis**。编写将库存加载到Redis的脚本。修改秒杀接口，使用Redis的 `DECR` 进行原子扣库存。
*   **目标**：完成核心秒杀逻辑，解决超卖问题。

### 第三周：性能优化与高阶特性

*   **Day 12-13**: **引入消息队列**。使用Redis List结构作为队列。修改秒杀接口，在Redis扣减库存成功后，将订单信息JSON序列化后推入队列，然后立即返回用户成功。
*   **Day 14-15**: **编写异步下单Worker**。一个独立Go程序，循环从Redis队列中阻塞拉取消息，然后执行创建订单等MySQL写操作。
*   **Day 16-17**: **实现限流中间件**。基于Redis实现IP限流。实现简单的秒杀地址隐藏（从静态URL改为动态生成）。
*   **目标**：完成异步化改造和限流，大幅提升系统抗压能力。

### 第四周：测试、部署与交付

*   **Day 18**: **压力测试与性能对比**。使用 **`wrk`** 或 **`ab`** 工具，对V1版（同步直接写DB）和最终版（Redis原子操作+消息队列）进行压测，记录QPS和错误率，对比分析性能差异。
*   **Day 19**: **编写单元测试**，对关键函数和Service层进行测试。完善错误处理和日志记录。
*   **Day 20-21**: **编写项目文档（README.md）**，包括架构说明、API接口文档、部署步骤。整理代码。
*   **Day 22**: **项目演示与验收**。
*   **目标**：交付一个高性能、高可用的秒杀服务，并具备完整的性能测试报告和文档。

## 六、质量要求

1.  功能质量：核心流程无逻辑漏洞，拓展功能满足需求，异常场景处理合理。
2.  性能质量：并发支撑能力、响应时间达标，无内存泄漏、死锁等问题。
3.  代码质量：遵循 Go 语言编码规范，函数职责单一，无冗余代码，关键逻辑有详细注释。
4.  数据质量：Redis 与 MySQL 数据最终一致，无超卖、少卖、订单丢失情况。

## 七、风险与应对措施

1.  风险 1：Redis 库存与 MySQL 数据不一致。应对措施：定期同步 Redis 库存至 MySQL，订单写入失败时触发库存回滚机制。
2.  风险 2：消息队列消息堆积。应对措施：设置消息持久化，动态扩容 Worker Goroutine 数量，监控堆积量并及时告警。
3.  风险 3：高并发下 Redis 性能瓶颈。应对措施：采用 Redis 集群分担压力，优化 Lua 脚本执行效率，减少 Redis 操作次数。
